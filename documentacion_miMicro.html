1. Descripción del Proyecto

El Problema

El transporte público en Santa Cruz de la Sierra opera de manera informal, generando estas problemáticas para los usuarios:

  • Incertidumbre total: No se sabe cuándo llegará el siguiente micro ni si tendrá lugar disponible.
  • Cambios de ruta sin aviso: Los choferes alteran recorridos por tráfico o decisión propia.
  • Pagos ineficientes: Solo efectivo, genera conflictos por cambio y riesgos de robo.
  • Vacío tecnológico: Aplicaciones globales como Google Maps no cubren las rutas informales locales.

La Solución — miMicro

miMicro es una plataforma web con tres actores principales que se complementan:

🧑 Pasajero

Ve el mapa en tiempo real con la posición de los micros, consulta tiempos de llegada (ETA), gestiona su tarjeta de transporte digital y recibe notificaciones de desvíos.

🚌 Chofer

Transmite su posición GPS en tiempo real, reporta el nivel de ocupación del micro y envía alertas de desvíos o incidentes a los pasajeros de su ruta.

🛠 Administrador

Gestiona usuarios (cambio de roles), administra Líneas de Micro (grupos de micros con ruta trazada en mapa), registra micros dentro de cada línea, y consulta estadísticas y logs de seguridad del sistema.

Flujo de comunicación GPS (tiempo real)

1. Chofer emite: navigator.geolocation.watchPosition() → frontend llama emitirGPS(microId, lat, lng, vel) → Socket.IO evento gps_update
2. Backend recibe: Escribe en Redis gps:micro:{id} con TTL 30s → emite micro_moved a sala ruta:{rutaId}
3. Pasajero recibe: Suscripto via join_ruta → escucha micro_moved → actualiza marcador en el mapa
4. Al parar: Chofer emite gps_stop → backend borra Redis key → emite micro_offline → mapa quita el marcador

// Pseudocódigo Dijkstra implementado
priority_queue<pair<double,int>, vector, greater> pq;
map<int,double> distancias;  // inicializadas a infinito
map<int,int>    previos;     // para reconstruir el camino
pq.push({0.0, origen});
distancias[origen] = 0.0;
while (!pq.empty()) {
    auto [dist, u] = pq.top(); pq.pop();
    for (auto& arista : grafo.obtenerVecinos(u)) {
        double nueva = dist + arista.peso;
        if (nueva < distancias[arista.destino]) {
            distancias[arista.destino] = nueva;
            previos[arista.destino] = u;
            pq.push({nueva, arista.destino});
        }
    }
}

// Fórmula Haversine implementada
R = 6371.0  // radio Tierra en km
dLat = (lat2 - lat1) * π / 180
dLng = (lng2 - lng1) * π / 180
a = sin(dLat/2)² + cos(lat1) * cos(lat2) * sin(dLng/2)²
c = 2 * atan2(√a, √(1−a))
distancia = R * c

    id               SERIAL PRIMARY KEY,

6. Backend API — Módulos y Endpoints

El servidor corre en http://localhost:3000. Documentación interactiva disponible en http://localhost:3000/docs (Swagger UI generado automáticamente por FastAPI).

Autenticación

Los endpoints protegidos requieren el header: Authorization: Bearer <token_JWT>
El token JWT contiene: { "sub": userId, "rol": "admin|chofer|pasajero", "nombre": "...", "exp": timestamp }
El cliente decodifica el payload con JSON.parse(atob(token.split('.')[1])) — sin librerías externas.

7. Frontend — Pantallas por Rol

La interfaz usa HTML5 + CSS3 + JavaScript puro, servida directamente por FastAPI en http://localhost:3000. Diseño dark mobile-first. Cada rol accede a su propio conjunto de páginas protegidas por JWT. La navegación se inyecta dinámicamente por layout.js según el rol.

Arquitectura del frontend — módulos compartidos

  • auth.js — IIFE superior inyecta meta PWA (manifest, apple-touch-icon, theme-color) y registra el Service Worker en cada página. Luego Auth.initPage(rol) verifica JWT y redirige. Tema por defecto: dark. escapeHtml(str) previene XSS.
  • api.js — Api.get/post/patch/del(url, body) agrega Authorization: Bearer automáticamente. El método DELETE es Api.del(), NO Api.delete().
  • layout.js — Layout.init(rol) bifurca por rol: admin recibe el sidebar lateral de escritorio; pasajero y chofer reciben mobile header (marca + toggles) + bottom nav (barra inferior con 4 íconos). Agrega clase has-bottom-nav al body para el layout CSS.
  • socket.js — Singleton de Socket.IO compartido entre páginas.

Excepción: la pantalla splash (/splash/) no carga auth.js. Tiene los meta PWA estáticamente y registra el SW inline. Se auto-redirige tras 2.5 segundos según el JWT en localStorage.

Enriquecimiento con micros activos (Redis)

Después de calcular el camino, routingRoutes.py consulta Redis para cada tramo de tipo micro. Itera todas las claves gps:micro:*, filtra por linea_id y calcula cuál micro activo está más cerca al origen del pasajero (fórmula Haversine). Devuelve distancia y ETA estimado basado en la velocidad real del micro (o 20 km/h por defecto).

Detalle técnico — tie-breaking en Dijkstra

El heap de Dijkstra usa tuplas (costo, seq, nodo) donde seq = next(itertools.count()). Esto es necesario porque Python 3 no puede comparar directamente str con tuple cuando dos caminos tienen el mismo costo — el número de secuencia garantiza orden total y evita el TypeError: '<' not supported between instances of 'str' and 'tuple'.
